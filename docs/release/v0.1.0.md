# iqdb-ivf v0.1.0 — Scaffold

**The repository bootstrap.** v0.1.0 establishes the structure, tooling, and quality gates the IVF index is built on — no domain logic yet. It exists so the implementation lands one verified step at a time onto a foundation that already enforces the iQDB engineering standard.

## What is iqdb-ivf?

The inverted-file (IVF) clustered nearest-neighbor index of the iQDB vector database. It partitions the vector space into clusters with k-means and, at query time, searches only the most relevant clusters — trading a small, tunable amount of recall for a large drop in work scanned. That makes it the complement to HNSW: more memory-efficient at very large scale and with more predictable latency. It ships IVF-Flat first (vectors stored as-is within clusters); IVF-PQ (quantized within clusters) layers on `iqdb-quantize`. It sits above `iqdb-types`, `iqdb-distance`, and `iqdb-index`, and integrates with `iqdb-filter` for filtered cluster search.

## What's next

- **v0.2.0 — k-means training + assignment + IVF-Flat search.** The hard part, not deferred: deterministic k-means training over a representative sample, vector-to-cluster assignment, and the IVF-Flat search path that probes a configurable number of clusters — property-tested against an exact full-scan oracle, with the "train before use" UX cliff documented loudly.
- **v0.3.0 — probe tuning + retrain + filtered cluster search.** The recall/latency dial, in-place retraining as the distribution drifts, and filtered search via `iqdb-filter`.
